# Phase 4: Sub-project D ‚Äî Formatting Bake-off (Formualizer vs IronCalc)

## Overview

Sub-project D (functional_spec ¬ß6.D, architecture ¬ß6, ¬ß1.1 bake-off) is the
**formatting/metadata differentiator** of the two-engine bake-off. It answers, per
engine (Formualizer and IronCalc) and head-to-head:

- **What formatting/metadata does the engine expose** ‚Äî row/col sizes, bold/italic,
  font size, fills, borders, number formats, merges, conditional formatting ‚Äî on
  **read**, on **write**, and across a **load ‚Üí edit ‚Üí save ‚Üí reload round-trip**?
- **What FreeCell formatting model** does that imply: native (via the engine / umya)
  vs a **side-table keyed by cell** vs a **custom Arrow-backed store**? Plus the
  next-best alternative and the load‚Üíedit‚Üísave verdict per engine.

The Sub-project A gate already pinned the decisive fact: **Formualizer 0.7.0 surfaces
`style: None` on every read path** (calamine `styles=false`; umya `read_cell` hard-codes
`style: None`), and `Workbook` does **not retain** the underlying umya `Spreadsheet`
after load ‚Äî `to_xlsx_bytes()` builds a *fresh* umya file from engine values/formulas,
so styles cannot ride through it. IronCalc, by contrast, exposes a real `Style` on
`get_style_for_cell` / `set_cell_style` and has native xlsx style I/O.

This phase **verifies** those claims with small passing tests (not vibes), probes the
concrete attribute matrix, and designs FreeCell's formatting store. Per the human's
note, the **exhaustive style-roundtrip-fidelity matrix is deferred to Round 2** ‚Äî this
phase does the design + capability probe, not an exhaustive fidelity sweep.

Guardrails: foreground-only (`timeout` on every command), operate strictly inside
`experiments/03-formatting/`, read-only use of `shared/`/`specs/`/`00‚Äì02`, never
`git add -A`, never commit.

## Layout

```
experiments/03-formatting/
  formualizer/    # probe crate: CellData styles gap; umya-direct style read/write; to_xlsx_bytes drop
  ironcalc/       # probe crate: Style read/write; row/col sizes; xlsx style round-trip
  results/        # machine-readable capability matrix (JSON) + human summary.md
  findings.md     # ¬ß5.2 head-to-head comparison + recommended FreeCell formatting model
```

Two independent Cargo crates (no workspace, per experiments/README.md). Each depends
on `datagen` by relative path (`../../shared/datagen`) so the neutral `CellFormat`
(bold, italic, highlight fill, h_align) is the common vocabulary the probes map onto
each engine. `bench_util` is **not** needed (no perf numbers here; this is a
capability probe, not a benchmark) ‚Äî but each result JSON is env-stamped by hand-rolled
small serde structs written to `results/`.

## Steps

### 1. `formualizer/` probe crate

- `Cargo.toml`: `name = "formualizer_formatting"`, edition 2021, `publish = false`,
  license MIT OR Apache-2.0. Deps:
  - `datagen = { path = "../../shared/datagen" }`
  - `formualizer = { version = "0.7", default-features = false, features = ["eval","parse","workbook","calamine","umya"] }`
    (need `umya` for `to_xlsx_bytes` + a direct umya style path; `calamine` for read).
  - `umya-spreadsheet = "2.3"` ‚Äî **direct** dependency: the style path Formualizer
    does not expose. This is the crux finding (Formualizer's own `UmyaAdapter` has a
    private inner `Spreadsheet` with no accessor, so we must own the umya workbook).
  - `serde` + `serde_json` for the results JSON.
- `src/lib.rs`: thin, documented helpers + a `capability matrix` builder:
  - `build_styled_xlsx_bytes() -> Vec<u8>` ‚Äî builds a small styled workbook **via umya
    directly** (bold, italic, font size, fill color, number format, a merge, row height,
    col width) and writes to bytes with `umya_spreadsheet::writer::xlsx::write_writer`.
    (datagen has no `.xlsx` writer ‚Äî xlsx generation was deferred to the file/format
    sub-projects ‚Äî so we generate the styled fixture from committed umya code, satisfying
    ¬ß5.3 "inputs generated by committed code".)
  - `load_into_formualizer(bytes) -> Workbook` ‚Äî `CalamineAdapter::open_bytes` +
    `Workbook::from_reader(.., LoadStrategy::EagerAll, WorkbookConfig::default())`.
  - `load_into_umya(bytes) -> umya_spreadsheet::Spreadsheet` ‚Äî `reader::xlsx::read_reader`.
  - `read_format_via_umya(&Spreadsheet, sheet, "A1") -> NeutralFormat` ‚Äî maps umya
    `Cell::get_style()` (font bold/italic/size, fill fg argb, number format code) into a
    neutral struct comparable to datagen's `CellFormat`.
  - a `FormattingCaps`/matrix producer that records, per attribute √ó {read, write,
    roundtrip}, `Native | ViaUmya | None` for Formualizer, serialized to
    `results/formualizer/capabilities.json`.
- `tests/probe.rs` ‚Äî passing assertions (verify, don't assume):
  - `celldata_style_is_none_on_read`: load the styled xlsx into a Formualizer
    `Workbook`; read a bold cell back through the `CellData` read path (write_range /
    read helpers) and assert `style` is `None` ‚Äî regression-lock the gap.
  - `values_survive_but_styles_dropped_through_to_xlsx_bytes`: build a Formualizer
    workbook with a value, `to_xlsx_bytes()`, reload via umya, assert the **value
    survives** but there is **no bold** (style dropped) ‚Äî proving `to_xlsx_bytes` is a
    values/formulas-only pipe.
  - `styles_readable_via_umya_directly`: load the same styled bytes into a
    `umya_spreadsheet::Spreadsheet`, assert bold/italic/fill/number-format are all
    readable ‚Äî proving the umya-direct path is the style source of truth.
  - `umya_style_edit_survives_roundtrip`: load ‚Üí set a cell bold + fill via umya ‚Üí
    write bytes ‚Üí reload via umya ‚Üí assert bold+fill survive (the load‚Üíedit‚Üísave verdict
    for the umya-direct path).
  - `workbook_does_not_retain_umya` (doc-locked): assert `UmyaAdapter` exposes no inner
    `Spreadsheet` accessor by relying only on the public API (documented in module docs;
    the behavioural proof is the dropped-style test above).

### 2. `ironcalc/` probe crate

- `Cargo.toml`: `name = "ironcalc_formatting"`, deps `datagen` (path),
  `ironcalc = "0.7"`, `ironcalc_base = "0.7"`, `serde`+`serde_json`. (Mirror the pinning
  note from 02: crate names use the UNDERSCORE `ironcalc_base`.)
- `src/lib.rs`: helpers + matrix builder:
  - `styled_model() -> Model<'static>` ‚Äî a small model with bold/italic/fill/number
    format set via `get_style_for_cell` + mutate + `set_cell_style`, plus
    `set_row_height`/`set_column_width` and (if API allows) a merge.
  - `read_format(&Model, sheet, row, col) -> NeutralFormat` ‚Äî maps IronCalc `Style`
    (`font.b`, `font.i`, `font.sz`, `fill.fg_color`, `num_fmt`) to the neutral struct.
  - `roundtrip_via_xlsx(model) -> Model` ‚Äî `ironcalc::export::save_to_xlsx` (or the
    bytes variant) then `ironcalc::import::load_from_xlsx_bytes` to reload, so styles
    cross a real xlsx boundary.
  - matrix producer ‚Üí `results/ironcalc/capabilities.json`.
- `tests/probe.rs`:
  - `styles_read_and_write_natively`: set bold+italic+fill+num_fmt via
    `set_cell_style`, read back via `get_style_for_cell`, assert all present.
  - `row_col_sizes_settable`: `set_row_height` / `set_column_width` then read back
    (via getter or the persisted xlsx) and assert.
  - `styles_survive_xlsx_roundtrip`: styled model ‚Üí save to xlsx bytes ‚Üí reload ‚Üí
    assert bold/italic/fill/num_fmt survive (the IronCalc load‚Üíedit‚Üísave verdict).
    If a specific attribute (e.g. conditional formatting) is not exposed via the public
    API, the test documents it as `None`/`Unverified` rather than asserting falsely.
  - `merges_if_supported` / `conditional_formatting_probe`: assert whatever the public
    API actually exposes; record `Unverified` in the matrix where the crate has no
    public surface, with a source pointer.

### 3. `results/`

- `results/formualizer/capabilities.json` and `results/ironcalc/capabilities.json`:
  env-stamped (Rust version, engine version, date `2026-07-01`) capability matrices
  (attribute ‚Üí {read, write, roundtrip} ‚Üí status).
- `results/summary.md`: the head-to-head table, generated/curated from the two JSONs.

### 4. `findings.md` (¬ß5.2 headings)

Rewrite the stub into: **Questions**, **What was done** (crate pointers + reproduce
commands), **Results / evidence** (the attribute √ó {read/write/roundtrip} √ó
{Formualizer, IronCalc} table, backed by the passing probes), **Conclusion** (direct
per-engine answer incl. "we couldn't determine X because Y" for deferred/unexposed
items), **Recommended design + next-best alternative** (native-via-engine/umya vs
side-table-keyed-by-cell vs custom-Arrow-backed store, with the load‚Üíedit‚Üísave verdict
per engine), **Risks / open questions**.
